# Telegram User-Defined Modes Design

## Summary

Change note for this task: `Breaking`.

Replace the fixed backend-owned Telegram focus mode enum with user-defined mode resources. A user can create, update, delete, and activate their own mode names, each mapped to a Telegram `emojiStatusDocumentId`. The backend keeps supporting a separate `defaultEmojiStatusDocumentId` for the "no active mode" state.

This change removes the old fixed-mode contract such as `personal`, `airplane`, `sleep`, and similar values. Existing stored values are preserved during migration, but after migration they are treated as plain user-defined strings instead of special backend-known modes.

## Goals

- Let each user create and manage their own Telegram mode names.
- Support a large number of user-defined mode mappings, for example dozens of entries per user.
- Keep the activation model simple: at most one mode can be active at a time.
- Preserve `defaultEmojiStatusDocumentId` as the fallback emoji when no mode is active.
- Keep the automation endpoint simple so iOS Shortcuts can continue sending only `mode` and `active`.

## Non-Goals

- Supporting multiple simultaneously active modes.
- Preserving backward compatibility with the fixed enum API.
- Keeping resolved mappings in the settings response as a derived map.
- Auto-creating modes from automation updates.

## Domain Model

### Telegram Mode Resource

Introduce a user-owned `telegram_mode` resource with:

- `id`
- `user_id`
- `mode_key`
- `emoji_status_document_id`
- `created_at`
- `updated_at`

Semantics:

- `mode_key` is a user-defined string, not an enum.
- `mode_key` is unique per user.
- `emoji_status_document_id` is the Telegram emoji status document ID that should be applied when this mode is active.

### Active Mode

Replace the current "many focus states with one or more active flags" model with a single nullable active mode reference stored on `telegram_account_connection`.

Add:

- `active_mode_id BIGINT NULL REFERENCES telegram_mode (id)`

Semantics:

- `NULL` means the user currently has no active mode.
- a non-null value means exactly one mode is active for that user.
- the referenced mode must belong to the same user as the connection row.

## Persistence Changes

### New Table

Create:

- `telegram_mode`

Suggested schema:

```sql
CREATE TABLE telegram_mode (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id BIGINT NOT NULL REFERENCES auth_user (id),
    mode_key VARCHAR(64) NOT NULL,
    emoji_status_document_id VARCHAR(64) NOT NULL,
    created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT telegram_mode_user_mode_unique UNIQUE (user_id, mode_key)
);
```

### Existing Table Change

Alter:

- `telegram_account_connection`

Add:

- `active_mode_id`

### Legacy Data Migration

Current legacy tables:

- `telegram_focus_mapping`
- `telegram_focus_state`

Migration strategy:

1. Create `telegram_mode`.
2. Add `active_mode_id` to `telegram_account_connection`.
3. Copy every legacy mapping row into `telegram_mode`.
4. For each user, resolve the currently active legacy mode and point `telegram_account_connection.active_mode_id` to the migrated `telegram_mode` row.
5. Preserve legacy names like `personal`, `sleep`, and `do_not_disturb` as plain `mode_key` strings.
6. Drop legacy tables after all application code has been moved to the new model.

This preserves current user data while intentionally changing the contract from backend-known modes to user-defined ones.

## HTTP API Design

### `GET /api/profile/me/telegram`

Keep the endpoint, but change the response contract.

Response should include:

- `connected`
- `connectionStatus`
- `telegramUser`
- `pendingAuth`
- `automationToken`
- `defaultEmojiStatusDocumentId`
- `activeFocusMode`
- `modes`

Response changes:

- `effectiveFocusMode` is removed and replaced by `activeFocusMode: string | null`
- `activeFocusModes` is removed because only one mode can be active
- `resolvedEmojiMappings` is removed
- `modes` is added as an array of mode resources

Suggested `modes` item shape:

```json
{
  "mode": "gym",
  "emojiStatusDocumentId": "1007"
}
```

### `PUT /api/profile/me/telegram/focus-settings`

Keep the endpoint only for default no-mode status settings.

Request:

```json
{
  "defaultEmojiStatusDocumentId": "7000"
}
```

Breaking change:

- remove `mappings` from the request contract

### `GET /api/profile/me/telegram/modes`

New endpoint that returns the current user's mode list.

Response:

```json
{
  "modes": [
    {
      "mode": "work",
      "emojiStatusDocumentId": "1001"
    },
    {
      "mode": "gym",
      "emojiStatusDocumentId": "1007"
    }
  ]
}
```

### `POST /api/profile/me/telegram/modes`

New endpoint that creates a mode.

Request:

```json
{
  "mode": "work",
  "emojiStatusDocumentId": "1001"
}
```

Behavior:

- trims and validates `mode`
- rejects blank `mode`
- rejects duplicate `mode` per user with `409`

### `PATCH /api/profile/me/telegram/modes/{mode}`

New endpoint that updates an existing mode.

Request:

```json
{
  "newMode": "deep_work",
  "emojiStatusDocumentId": "1002"
}
```

Behavior:

- `newMode` is optional
- `emojiStatusDocumentId` is optional
- at least one field must be present
- if the source mode does not exist, return `404`
- if `newMode` conflicts with another user mode, return `409`
- if the updated mode is currently active and renamed, the active mode association must continue pointing to the same row

### `DELETE /api/profile/me/telegram/modes/{mode}`

New endpoint that deletes an existing mode.

Behavior:

- if the mode does not exist, return `404`
- if the mode is currently active, clear the active mode first
- delete the mode row
- after deleting the active mode, Telegram should fall back to `defaultEmojiStatusDocumentId` on the next automation sync or immediate sync path used by the endpoint implementation

### `POST /api/telegram/focus-updates`

Keep the endpoint shape but change `mode` to a plain string.

Request:

```json
{
  "mode": "work",
  "active": true
}
```

Behavior:

- authenticate using the personal automation token as before
- `mode` is treated as a user-owned string
- when `active = true`:
  - find the user's mode by `mode`
  - if not found, return a stable `404` error
  - set it as the single active mode
  - apply its `emojiStatusDocumentId` to Telegram
- when `active = false`:
  - if the provided mode is the current active mode, clear the active mode and apply `defaultEmojiStatusDocumentId`
  - if the provided mode is not currently active, do not change the active mode

Response:

- keep `applied`
- replace `effectiveFocusMode` with `activeFocusMode`
- keep `appliedEmojiStatusDocumentId`

## Validation Rules

### Mode Key

Server-side normalization:

- trim leading and trailing whitespace
- reject blank values
- maximum length: `64`

Comparison:

- case-sensitive

Rationale:

- case-sensitive strings avoid hidden normalization rules and surprising collisions

### Emoji Status Document ID

Server-side normalization:

- trim leading and trailing whitespace
- reject blank values on create
- allow omission on patch only when another field is present

## Error Model

Add stable error codes for mode resource operations.

Suggested codes:

- `TELEGRAM_MODE_NOT_FOUND`
- `TELEGRAM_MODE_ALREADY_EXISTS`
- `TELEGRAM_MODE_INVALID`

HTTP behavior:

- `404` for missing mode resources or missing automation mode on activation
- `409` for duplicate create or rename conflicts
- `400` for invalid or blank mode values

Existing token/auth/connectivity errors stay unchanged.

## Application Layer Design

Add focused application services around the new resource and single-active-mode workflow.

Suggested responsibilities:

- mode CRUD service
- active mode sync/update service
- settings service for `defaultEmojiStatusDocumentId`
- query service assembling the new response shape

Controller rules:

- profile controller stays thin and delegates mode operations to application services
- automation controller stays thin and delegates token-authenticated activation/deactivation to application service

## Testing Strategy

### Unit Tests

Add or update tests for:

- creating a new mode
- rejecting duplicate mode creation
- updating emoji for a mode
- renaming a mode
- rejecting rename conflicts
- deleting a mode
- deleting the active mode clears active state
- activating a mode through automation update
- deactivating the active mode restores default emoji status
- deactivating a non-active mode is a no-op

### Integration Tests

Cover end-to-end flows:

1. create user-defined modes
2. fetch settings and verify `modes`
3. activate a created mode through `/api/telegram/focus-updates`
4. switch to another created mode
5. deactivate the active mode and verify fallback to `defaultEmojiStatusDocumentId`
6. delete the active mode and verify active state is cleared
7. verify unknown mode activation returns stable `404`

## Compatibility Notes

This is an intentional breaking change.

Frontend and iOS Shortcut clients must stop relying on backend-known fixed values such as:

- `personal`
- `airplane`
- `do_not_disturb`
- `reduce_interruptions`
- `sleep`

Clients must instead manage user-defined mode resources through the backend and send those exact user-created strings back to `/api/telegram/focus-updates`.
