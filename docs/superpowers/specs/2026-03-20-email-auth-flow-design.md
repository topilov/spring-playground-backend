# Email Auth Flow Design

## Summary

Add a complete authentication email flow to the existing Spring Boot project:

- `POST /api/auth/register`
- `POST /api/auth/forgot-password`
- `POST /api/auth/reset-password`
- email delivery for `welcome` and `reset-password`
- HTML email templates rendered with Thymeleaf

The implementation should stay intentionally simple and aligned with the current codebase: synchronous application-service orchestration, JPA persistence, Flyway migrations, and direct email sending through Spring Mail.

## Goals

- Support self-service registration over HTTP.
- Send a welcome email after a successful registration.
- Support password reset through a one-time reset token flow over HTTP.
- Render HTML emails with Thymeleaf templates.
- Keep the solution small, testable, and consistent with the current auth/profile architecture.

## Non-Goals

- Asynchronous email queues or outbox processing.
- Email verification or account activation.
- Rich audit history beyond password reset token state.
- Frontend pages for password reset handling.

## Existing Context

The project currently has:

- session-based authentication with `/api/auth/login` and `/api/auth/logout`
- `AuthUser` persisted in `auth_user`
- `UserProfile` persisted in `user_profile`
- JPA + Flyway + PostgreSQL
- integration tests built around `MockMvc` and Testcontainers

There is no current email infrastructure, registration flow, or password reset flow.

## Recommended Approach

Implement the feature as a minimal, production-shaped extension of the current auth module:

1. Add the new HTTP endpoints to `AuthController`.
2. Extend `AuthService` with registration and password reset orchestration.
3. Add a small mail subsystem responsible for rendering and sending Thymeleaf emails.
4. Persist password reset tokens in a dedicated table.
5. Cover the flow with integration tests and focused service tests.

This approach avoids premature infrastructure while still delivering a complete end-to-end feature set.

## Endpoint Policy

Public endpoints:

- `GET /actuator/health`
- `GET /api/public/ping`
- `POST /api/auth/login`
- `POST /api/auth/logout`
- `POST /api/auth/register`
- `POST /api/auth/forgot-password`
- `POST /api/auth/reset-password`

Authenticated endpoints:

- `GET /api/profile/me`
- `PUT /api/profile/me`
- any future user-account endpoints that operate on an authenticated session rather than a one-time reset token

Default policy:

- all endpoints are authenticated unless explicitly listed as public
- the new registration and password reset endpoints stay public because they must work before a user has a session

## API Design

### `POST /api/auth/register`

Request body:

```json
{
  "username": "demo",
  "email": "demo@example.com",
  "password": "strong-password"
}
```

Behavior:

- validate request fields
- reject duplicate username and duplicate email
- encode the password with `PasswordEncoder`
- create `AuthUser`
- create a default `UserProfile`
- send a `welcome` email after successful persistence
- do not create an authenticated session and do not auto-login the user
- return a success response with basic created-user data

Suggested response shape:

```json
{
  "userId": 2,
  "username": "demo",
  "email": "demo@example.com"
}
```

### `POST /api/auth/forgot-password`

Request body:

```json
{
  "email": "demo@example.com"
}
```

Behavior:

- always return the same success response, whether or not the email exists
- if the email belongs to a user:
  - invalidate older active reset tokens for that user
  - generate a new secure one-time token
  - store only a hash of the token in the database
  - send a `reset-password` email containing a reset URL with the raw token

Suggested response shape:

```json
{
  "accepted": true
}
```

### `POST /api/auth/reset-password`

Request body:

```json
{
  "token": "raw-reset-token",
  "newPassword": "new-strong-password"
}
```

Behavior:

- hash the incoming token
- find an active matching token record
- reject missing, expired, or already-used tokens
- update the user's password hash
- mark the token as used
- invalidate remaining active tokens for the same user
- return a success response

Suggested response shape:

```json
{
  "reset": true
}
```

## Data Model

Add a new Flyway migration for `password_reset_token`.

Suggested schema:

```sql
CREATE TABLE password_reset_token (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id BIGINT NOT NULL REFERENCES auth_user (id),
    token_hash VARCHAR(64) NOT NULL UNIQUE,
    expires_at TIMESTAMP WITH TIME ZONE NOT NULL,
    used_at TIMESTAMP WITH TIME ZONE,
    created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_password_reset_token_user_id
    ON password_reset_token (user_id);
```

### Token Handling

- Generate a cryptographically secure random token.
- Send the raw token only through email.
- Persist only a deterministic hash of the token, for example SHA-256 hex.
- Consider a token active when:
  - `used_at IS NULL`
  - `expires_at > now`

### Token Lifetime

Default recommendation: 30 minutes.

This should be configurable through application properties.

## Mail Subsystem

Introduce a dedicated mail package, for example `me.topilov.springplayground.mail`.

Core components:

- `MailProperties`
  - `from`
  - `app-name`
  - `public-base-url`
  - `reset-password-path`
  - `password-reset-ttl`
- `EmailTemplate` enum or sealed representation
  - `WELCOME`
  - `RESET_PASSWORD`
- `EmailService`
  - common interface for sending typed emails
- `ThymeleafEmailService`
  - renders templates through `SpringTemplateEngine`
  - sends MIME HTML email through `JavaMailSender`

### Templates

Store templates in:

- `src/main/resources/templates/mail/welcome.html`
- `src/main/resources/templates/mail/reset-password.html`

Template variables:

`welcome.html`

- `appName`
- `username`

`reset-password.html`

- `appName`
- `username`
- `resetUrl`
- `expiresInMinutes`

Plain HTML templates are sufficient; no separate text variant is required for this change.

### Reset Link URL

The reset link in the email should target the user-facing application URL, not the backend service URL.

- `public-base-url` should point to the frontend or other public entrypoint the user opens in a browser
- `reset-password-path` should be a frontend route such as `/reset-password`
- the final link is composed from `public-base-url`, `reset-password-path`, and the raw token

Example:

```text
https://app.example.com/reset-password?token=...
```

## Application Flow

### Registration Flow

1. Controller validates `RegisterRequest`.
2. Service checks uniqueness constraints.
3. Service persists `AuthUser`.
4. Service persists matching `UserProfile`.
5. Transaction commits the new user and profile.
6. Service attempts to send the welcome email after commit.
7. Controller returns the success payload.

Registration does not auto-login the user. After a successful registration, the client must call the normal login endpoint to create a session.

Mail failure policy for registration:

- user creation and profile creation happen in one transaction
- welcome email sending happens after the transaction commits
- if welcome email delivery fails, the created account remains valid
- the endpoint still returns success because account creation is the primary contract of the endpoint
- the failure should be logged clearly so it can be diagnosed

### Forgot Password Flow

1. Controller validates `ForgotPasswordRequest`.
2. Service looks up user by email.
3. If absent, return accepted response without side effects.
4. If present:
   - invalidate active tokens
   - create new token
   - persist hashed token metadata in one transaction
   - commit token changes
   - compose reset URL from `public-base-url`, `reset-password-path`, and raw token
   - attempt to send reset email after commit
5. Return accepted response.

Mail failure policy for forgot-password:

- the endpoint always returns the same accepted response, whether or not the email exists
- reset-token persistence happens before the email attempt
- if reset email delivery fails, the response still stays generic to avoid leaking whether the email belongs to an account
- the failure should be logged clearly so the issue is visible operationally
- a later forgot-password request may issue a fresh token and invalidate prior active tokens

### Reset Password Flow

1. Controller validates `ResetPasswordRequest`.
2. Service hashes the provided token.
3. Service loads a matching token record.
4. Service checks expiration and usage state.
5. Service updates the user's password hash.
6. Service marks the token used and invalidates siblings.
7. Return success response.

Transaction policy for reset-password:

- password update and token state changes happen in one transaction
- if password persistence fails, the token must remain unused
- there is no outgoing email in this flow

## Validation and Error Handling

### Request Validation

- `username`: non-blank, sensible max length aligned with DB schema
- `email`: non-blank, valid email format, max length aligned with DB schema
- `password`: non-blank, minimum length, sensible maximum
- `token`: non-blank

### Domain Errors

Add explicit exceptions for:

- duplicate username
- duplicate email
- invalid password reset token
- expired password reset token

Map these to stable HTTP responses, likely:

- duplicate username/email: `409 Conflict`
- invalid or expired reset token: `400 Bad Request`

`forgot-password` should not reveal whether a user exists for the provided email.

## Security Considerations

- Use `PasswordEncoder` consistently for registration and reset.
- Never log raw reset tokens.
- Never store raw reset tokens in the database.
- Return a generic success response for `forgot-password`.
- Invalidate previous active reset tokens when issuing a new one.
- Invalidate remaining active reset tokens once a password reset succeeds.

## Configuration

Add Spring Mail and Thymeleaf dependencies.

Extend configuration with:

```yaml
spring:
  mail:
    host: ${MAIL_HOST:localhost}
    port: ${MAIL_PORT:1025}
    username: ${MAIL_USERNAME:}
    password: ${MAIL_PASSWORD:}
    properties:
      mail.smtp.auth: ${MAIL_SMTP_AUTH:false}
      mail.smtp.starttls.enable: ${MAIL_SMTP_STARTTLS:false}

app:
  mail:
    from: ${MAIL_FROM:no-reply@example.com}
    app-name: ${MAIL_APP_NAME:Spring Playground}
    public-base-url: ${APP_PUBLIC_BASE_URL:http://localhost:3000}
    reset-password-path: ${APP_RESET_PASSWORD_PATH:/reset-password}
    password-reset-ttl: PT30M
```

`application-test.yml` should provide safe test defaults.

## Testing Strategy

Follow TDD.

### Integration Tests

Add endpoint-level tests covering:

- successful registration creates user and profile and triggers welcome email
- duplicate registration is rejected
- forgot password returns accepted for both existing and non-existing email
- forgot password for existing user persists a token and triggers reset email
- reset password with a valid token updates the stored password hash
- reset password rejects expired or invalid token

### Service-Level Tests

Add focused tests for the mail layer and token behavior where integration coverage alone would be too coarse:

- Thymeleaf template rendering includes expected fields
- email sender builds HTML MIME messages correctly
- token invalidation rules behave as expected

### Test Doubles

- prefer `@MockBean` or equivalent test double for `JavaMailSender`
- assert recipient, subject, and presence of expected HTML content
- avoid external SMTP dependency in tests

## Implementation Notes

- Keep package boundaries aligned with current modules.
- Reuse the existing auth service instead of introducing an event bus.
- Use a dedicated repository for password reset tokens.
- Prefer explicit DTOs for each request and response.
- Create profile data alongside registration to preserve the project's current user/profile consistency model.

## Risks

- Synchronous email sending adds latency to registration and forgot-password requests.
- Mail delivery failures can fail user-facing flows unless explicitly handled differently.
- Template rendering can become brittle if tests assert too much exact HTML.

These risks are acceptable for the current scope and simpler than adding asynchronous infrastructure.

## Rollout

Implement in small TDD slices:

1. registration endpoint and persistence
2. welcome email sending
3. forgot-password token issuance and email
4. reset-password token consumption and password update
5. refactor shared mail and token helpers
